----------------------------------------------------------------
SECTION 1: CURRENT STATE
----------------------------------------------------------------

The PvP match system is entirely unauthenticated. Any client
that knows a matchId can send actions on behalf of either player.

HOW MATCHES WORK TODAY:
  - POST /api/match/create  → body: { playerName }
      response: { matchId, playerId: "p1" }

  - POST /api/match/join    → body: { matchId, playerName }
      response: { matchId, playerId: "p2" }

  - POST /api/match/[id]/action → body: { playerId: "p1"|"p2", type: ActionType }
      response: { state: MatchState }

The client is trusted to self-report which playerId it is.
The server does no verification. The only protection is the
turn-check: "if (state.turn !== playerId) throw error". But
this only prevents out-of-turn actions — it does not prevent
a rogue client from impersonating a player on their own turn.

THE VULNERABILITY:
  1. Alice creates a match. Server assigns her "p1". matchId: "XYZ789".
  2. Bob joins. Server assigns him "p2".
  3. Eve discovers the matchId "XYZ789" (e.g., from the URL bar,
     network tab, shared link, or by guessing).
  4. It is Alice's turn (p1). Eve sends:
       POST /api/match/XYZ789/action
       body: { playerId: "p1", type: "HEAL" }
  5. The server executes the action as Alice. Alice never moved.
  6. Eve can also spoof Bob's moves when it is p2's turn.

No cryptographic token, session cookie, or identity check
exists on any match endpoint.

KEY FILES (CURRENT):
  - apps/web/lib/game-server/match.ts       → match state + combat logic
  - apps/web/lib/game-server/types.ts       → MatchState, PlayerState types
  - apps/web/app/api/match/create/route.ts  → creates match, assigns p1
  - apps/web/app/api/match/join/route.ts    → assigns p2
  - apps/web/app/api/match/[id]/action/route.ts → applies action (no auth)
  - apps/web/app/api/match/[id]/stream/route.ts → SSE stream (no auth)
  - apps/web/app/match/page.tsx             → frontend PvP UI
  - apps/web/lib/auth-session.ts            → existing auth utilities
  - apps/web/prisma/schema.prisma           → no Match model exists

STATE STORAGE TODAY:
  All match state lives in two in-memory Maps inside match.ts:
    const matches     = new Map<string, MatchState>();
    const subscribers = new Map<string, Set<SSECallback>>();
  These are lost on server restart. The Redis migration (Issue #2)
  will replace both Maps. Tokens will move to Redis as part of
  that migration — they live inside the MatchState JSON blob.

THE EXISTING AUTH SYSTEM (for reference):
  - Firebase Auth + session cookies for the main app.
  - verifyRequestAuth(req) checks session cookie or Bearer token.
  - This is used on /api/my-cards and admin routes only.
  - Match routes do not call verifyRequestAuth at all.

WHAT WE ARE NOT DOING:
  - We are NOT requiring a Firebase login to play PvP. The
    current design allows guest play (playerName only). We
    will keep that. Per-match tokens are separate from
    Firebase identity.

SECTION 2: TARGET STATE (WHAT WE WANT)
----------------------------------------------------------------

When a player creates or joins a match, the server generates a
cryptographically random token unique to that player's seat in
that match. The token is returned once and stored server-side
(in MatchState, which Redis will persist). Every subsequent
action request must include that token. The server validates
the token before processing any action.

NEW FLOW:
  - POST /api/match/create
      request:  { playerName: string }
      response: { matchId: string, playerId: "p1", token: string }

  - POST /api/match/join
      request:  { matchId: string, playerName: string }
      response: { matchId: string, playerId: "p2", token: string }

  - POST /api/match/[id]/action
      request:  { type: ActionType }
      header:   Authorization: Bearer <token>
      (playerId is no longer in the request body — the server
       resolves it from the token stored in MatchState)

  - GET /api/match/[id]/stream
      No auth required. Stream pushes PublicMatchState, which
      never contains tokens.

HOW TOKENS ARE STORED:
  Tokens are fields on MatchState. They are serialized as part
  of the match JSON blob wherever state is stored:
    - TODAY:    inside the in-memory matches Map
    - WITH REDIS: inside Redis key "match:{id}" (same JSON,
                  same fields — no separate token key needed)

  A token is a 32-byte random hex string (64 characters).
  The server stores one token per player seat: p1Token, p2Token.

WHAT CHANGES IN MatchState:
  Before:
    interface MatchState {
      matchId: string;
      status: "waiting" | "active" | "finished";
      players: { p1: PlayerState; p2: PlayerState | null };
      turn: PlayerId;
      log: string[];
      winner: PlayerId | null;
    }

  After:
    interface MatchState {
      matchId: string;
      status: "waiting" | "active" | "finished";
      players: { p1: PlayerState; p2: PlayerState | null };
      turn: PlayerId;
      log: string[];
      winner: PlayerId | null;
      p1Token: string;          // ← NEW: never sent to clients
      p2Token: string | null;   // ← NEW: null until p2 joins
    }

  A second type, PublicMatchState, is identical to MatchState
  except p1Token and p2Token are omitted. All responses to
  clients (GET, SSE, action response) use PublicMatchState.
  This stripping happens in a helper function, not in the
  individual routes, so it cannot be accidentally skipped.

WHAT CHANGES IN THE ACTION ENDPOINT:
  Before:
    body: { playerId: "p1" | "p2", type: ActionType }
    → server trusts the provided playerId

  After:
    header: Authorization: Bearer <token>
    body: { type: ActionType }
    → server calls resolvePlayerByToken(matchId, token)
    → resolvePlayerByToken reads MatchState and compares token
      to p1Token / p2Token
    → server uses the resolved playerId, never the client's claim

WHAT CHANGES ON THE FRONTEND:
  - After create/join, token is stored in localStorage (not
    sessionStorage — localStorage survives page refresh, which
    aligns with the Redis disconnect-recovery plan in Issue #2).
  - localStorage key: "matchSession" holds JSON:
      { matchId, playerId, playerName, token }
    This matches the reconnection object from match-persistence-plan.md
    with token added.
  - Every action fetch includes:
      headers: { "Authorization": "Bearer " + token }
  - On page load, if "matchSession" exists in localStorage, the
    app attempts to reconnect to that match (fetch match state,
    re-subscribe to SSE). The token is recovered from localStorage
    and action requests resume normally.
  - On match finish (winner !== null), clear localStorage:
      localStorage.removeItem("matchSession")

DISCONNECT RECOVERY WITH TOKENS:
  Because Redis keeps match state alive through server restarts,
  and because localStorage keeps the token alive through page
  refreshes, a player can fully recover a match:

    Player closes tab or server restarts:
      1. Player reopens tab.
      2. App reads localStorage → finds { matchId, token, ... }.
      3. App fetches GET /api/match/{matchId}.
      4. Match exists in Redis → app reconnects SSE → game resumes.
      5. On next action, the saved token is sent in the header.
      6. Server validates token from MatchState in Redis → OK.

  No re-authentication, no re-join. The token that was issued
  at create/join time remains valid for the life of the match.

STEP 1: Update Type Definitions

        return pub;
      }

    All routes call it before sending data to clients.

  NO REDIS DEPENDENCY. Safe to implement now.


STEP 2: Add Token Generation and Resolution to Match Logic
  File: apps/web/lib/game-server/match.ts

  - Import crypto (Node built-in):
      import crypto from "crypto";

  - Add generateToken():
      function generateToken(): string {
        return crypto.randomBytes(32).toString("hex");
      }

  - In createMatch(): generate p1Token, set p2Token to null.
    Store both in the initial MatchState object.

  - In joinMatch(): generate p2Token. Set it in the state before
    persisting/publishing.

  - Add resolvePlayerByToken():
      // [REDIS INTEGRATION POINT]
      // TODAY (in-memory): synchronous Map lookup.
      //   const match = matches.get(matchId);
      // WITH REDIS: this becomes async — await getMatch(matchId).
      // The logic inside is identical either way.
      function resolvePlayerByToken(
        matchId: string,
        token: string
      ): PlayerId | null {
        const match = matches.get(matchId); // stub: replace with getMatch()
        if (!match) return null;
        if (match.p1Token === token) return "p1";
        if (match.p2Token === token) return "p2";
        return null;
      }

  - Mark resolvePlayerByToken as async-ready: add a comment
    noting it must become async when Redis lands, since getMatch()
    will be async. The action route will need to await it.

  - Ensure createMatch() and joinMatch() call toPublicState()
    on any MatchState they return to their callers (routes).
    The raw MatchState with tokens stays internal.

  PARTIAL REDIS DEPENDENCY: The function body needs a one-line
  change when Redis lands (Map lookup → await redis.get).
  Mark with // [REDIS INTEGRATION POINT] comment.

STEP 5: Update Action Route (Core Security Change)
  File: apps/web/app/api/match/[id]/action/route.ts

  - Remove playerId from request body parsing.
  - Read Authorization header:
      const authHeader = req.headers.get("Authorization");
      if (!authHeader?.startsWith("Bearer ")) {

      }
      const token = authHeader.slice(7);

  - Resolve playerId server-side:

SECTION 5: SECURITY NOTES
----------------------------------------------------------------

TOKEN SCOPE:
  Tokens are match-scoped and player-scoped. A p1 token for
  match "ABC123" cannot be used in match "XYZ789" or to act
  as p2 in the same match. resolvePlayerByToken() checks both
  the match existence AND the specific seat.

TOKEN LIFETIME:
  TODAY:    Token lives as long as the in-memory match lives.
  WITH REDIS: Token lives as long as the Redis key lives (15-min
              TTL, refreshed on every action per the Redis plan).
              If the match expires from Redis, the token is also
              gone — clients will get 404 on reconnect and clear
              localStorage automatically.

TOKEN SECRECY:
  - Tokens are returned ONCE at create/join time.
  - Tokens are never in GET /api/match/[id] responses.
  - Tokens are never in SSE stream data.
  - toPublicState() is the single enforcement point for this.
  - localStorage is accessible to JavaScript on the same origin.
    This is acceptable for this trust model (we are not adding
    Firebase auth to matches). Same risk as any session token
    stored in localStorage.

WHAT THIS DOES NOT FIX:
  - It does not prevent two players sharing a device from sharing
    a localStorage token (physical access is out of scope).
  - It does not prevent the join race (someone sniping p2 slot).
    That is a separate invite-link concern.
  - It does not add rate limiting on actions.

WHAT THIS DOES FIX:
  - Eve cannot impersonate Alice even if she knows the matchId.
  - A browser can only act as the player it was assigned at
    create/join time. The server, not the client, resolves identity.
  - Survives disconnects and page refreshes — the token is
    recovered from localStorage and remains valid in Redis.

FUTURE CONSIDERATION: Firebase + Match Token Integration
----------------------------------------------------------------

NOT part of this feature. Tracked separately.

When we eventually require Firebase login to play PvP (e.g. to
award card rewards, track win/loss records, or gate match access
to registered users), the integration point would be:

  - Require a valid Firebase session (verifyRequestAuth) on the
    create and join routes before issuing a match token.
  - Store the player's firebaseUid alongside their match token
    in MatchState so the server can associate a match seat with
    a real user account.
  - On match finish, use the stored firebaseUid to look up the
    Prisma User record and apply rewards.

The match token mechanism itself does not need to change — it
still acts as the per-match identity for action validation.
Firebase auth would just become a prerequisite gate at
create/join time.

This is a separate feature from token-based impersonation
prevention and should be planned and implemented independently.
